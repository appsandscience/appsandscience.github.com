The MVCS architecture and file structure

Sencha Touch applications follow the Model-View-Controller (MVC) architecture. The architecture separates the user interface (view) from the structure of the data that the application serves (model) and the controller that handles inputs from the user. Sencha Touch uses an additional concept, that of a Store. A store describes how data are retrieved: from a file, from a web service, etc. In other words, Sencha Touch uses the MVCS architecture.

The organization of files in Sencha Touch is strict allowing developers to structure the application files in a hierarchical order. The structure of an application is shown in Figure 1.

Two files are on the root of the file tree: designer.html and designer.js. (The files are named designer because they are generated by Sencha’s IDE, called Ext Designer.) Under a directory called app there are four directories, one for each component of the MVCS architecture. The names of the directories are standard: controller, model, store, view. Under those directories the source code for each component is stored.

Making Sense of Sencha Touch: The CityBars App - Views

Feb 20th, 2012

In a previous post, I described the advantages of using Sencha Touch to develop applications for mobile platforms. In this post I would like to describe a few Sencha Touch concepts.

If you would like to see how a Sencha Touch application works, check out the kitchensink application that has been created by the developers of Sencha Touch. You will be able to preview the JavaScript code that generates these views and you will be surprised by the relative short length of the code, at least compared to that necessary in Objective-C.

Some additional digging around might show you that it’s not very easy to program in Sencha Touch. Although Sencha Touch is easier to pick up compared to Objective-C, especially for web developers, the platform is not very straightforward. Mostly because of the complicated terminology used in Sencha Touch, and the dearth of books on the subject: the Sencha Touch Cookbook is the only currently published book on Sencha Touch and it covers version 1.1, and not the current version 2.

This will be the first in a series of blog posts that aim to describe how to develop, build and deploy a mobile application using Touch 2. After you read this and the upcoming posts you will be able to

  • understand several concepts of Sencha Touch,
  • follow along with this and other screencasts created by Sencha
  • use Sencha’s Designer to create the application from scratch.

For all the posts I will use the CityBars application that is explained in this screencast and its source code is available here.

CityBars finds the location of the user and lists nearby bars. The application provides that same results that you would obtain using Yelp’s mobile application because the information about each bar is pulled from Yelp. In this post I will describe the components of the graphical user interface of CityBars. In future posts, I will write about the way that CityBars is pulling data from the Internet, how it finds the location of the user and how it responds to user taps.

The application contains two types of screens. One that shows the list of the bars and one that shows details about the bar. This is a Master-Detail type of application.

Common to both screens of the CityBars application is a navigation bar docked on the top of the screen. In Sencha Touch 2 terminology the navigation bar is an example of a NavigationView component. (Important: All Sencha Touch 2 terms will be written in bold from now on.) When an item below the navigation bar is tapped the navigation bar changes its title to match that of the tapped item and a Back button appears.

Below the navigation bar there are two screens or Views. The first one shows a list a Dataview.List of the bars in the area.

The second view is more complicated because it contains a tab panel (tabPanel) with two tabs: The first tab panel consists of two Container components. The first Container shows an image of the bar its name its address and its rating. The second Container is stacked below the first and contains two buttons: one that can be used to call the bar and one to that can be tapped to get more information about the bar. Both buttons are inactive in this version of the application.

The second panel consists of a Map component that displays the location of the bar on a Google map. Unfortunately, there is no simple correspondence between what a user would call a screen and what Sencha Touch calls a View. According to Sencha Touch, the first screen the list of bars contains two components: The NavigationView and below it a Container that displays the DataView.List or simply the list. Similarly the second screen contains a tabPanel with a tabBar on the top and two different tabs each of which contains one or more Container components.

I will now describe each screen of CityBars using Sencha Touch terminology. The descriptions below will be useful in a future post when I look at the application code.

The NavigationView component has a light theme as opposed to the default dark theme. There is a single item contained in the NavigationView with the title CityBars and that item is a List Container. The list Container is set to fit the entire space available to it with the following item: A 40x40 pixel image of the bar and next to it its name, its rating and its address. When a user taps on an item, the app displays a tab.Panel which is docked to the top of the screen. The tab.Panel looks darker than the navigation bar and its tabs are arranged horizontally and centered across the bar. By default the application displays the Contact tab which consists of the Contact Container (Figure 2). The Container in turn, contains several Container components itself.

The first Container contains two Components: One that shows a 100x100 pixel photo of the bar and a second that shows the bar name address and rating. The second Container packs in the center and stacks horizontally a Call Button a Spacer and a More Button. When the tab called Map is tapped the app shows the Map Container (Figure 3). The Container is titled Map and contains a single Map Component.

Creating an iPhone Application With JavaScript: The Sencha Touch Solution

Feb 13th, 2012

During the past few months, I spent countless hours studying Objective-C, the language used for iPhone and iPad app development. Just to give you an idea, the books and tutorials I have studied from include

  1. iPhone and iPad Apps for Absolute Beginners, iOS 5 Edition by Rory Lewis,

  2. iOS Programming: The Big Nerd Ranch Guide (2nd Edition), by Aaron Hillegass,

  3. Objective-C Programming: The Big Nerd Ranch Guide, by Aaron Hillegass,

  4. iPad and iPhone Application Development, by Paul Hegarty, and

  5. Video2Brain: Getting Started with iOS 5, by Robert Turrall.

All these books and tutorials taught me a lot about Objective-C but mostly that it is a very hard language to learn. In addition, Objective-C is only useful for developing iPhone/iPad applications. But how could I develop applications that can run on Android devices? Would I need to reprogram my apps in Java, the language used by Android? I was hoping that I could find a platform that could convert a single program into apps that can run on multiple operating systems (Apple, Android, even Blackberry) and multiple devices (smartphones, tablets, or the web). I was astonished to find out that many such languages exists (see the Wikipedia page on Mobile Development for an exhaustive list.)

Now my problem became which platform to choose. I experimented with several of them.

  • First, I worked with Canappi. (See my blog post for a detailed description.) Canappi uses a specialized language (MDSL) that can be used to generate Objective-C code. Unfortunately, the code that Canappi generates has includes deprecated statements, probably as a result of Canappi being the product of a single developer.

  • Then, I looked at Tiggzi. (See my blog post for a detailed description.) Tiggzi uses a GUI that helps you create iPhone and Android applications without any coding. Tiggzi is a useful tool, but it’s not free and open-source.

  • I also looked at Corona SDK, a platform that uses the scripting language Lua. Corona SDK is commonly used to develop 2D games, with several iPhone games on the app store, but not for business, data-driven applications.

  • I finally looked at ways of creating apps using HTML5 and its associated technologies, JavaScript and Cascading Style Sheets (CSS). Although I briefly looked at Dreamweaver, I was turned off by its price. It was at that point that I looked into Sencha Touch.

Sencha Touch is a JavaScript library built for the development of web applications that look native on mobile devices. After I read some of its documentation and reviewed the kitchensink application developed with Sencha Touch, I decided to spend some time to learn the platform. There are several advantages to Sencha Touch.

Sencha Touch

  1. Is free and fast (especially its latest version.)

  2. Simplifies the creation of nested views and the insertion of images on table entries.

  3. Works with Phonegap but it also allows native packaging.

  4. Uses the MVC (model-view-controller) architecture for code organization and reusability.

  5. Aims to support Windows phones in the future.

Unfortunately, Sencha Touch, has a steep learning curve, like Objective-C, but unlike Objective-C, there are not that many books that describe its use, apart from the very useful Sencha Touch Cookbook, by Ajit Kumar.

I hope to slowly learn the platform in order to be able to quickly and robustly develop data driven applications. Future blog post will describe my progress in using Sencha Touch.

Creating an iPhone Application Without Coding: The Tiggzi Solution

Feb 6th, 2012

For the past few weeks I have been experimenting with various tools that one can use to develop an iPhone/iPad application, without the need to program in Objective-C. In a previous blog post I described Canappi, a tool that uses a specialized language, MDSL, to generate Objective-C code. In this post, I will describe Tiggzi, a web service that creates native code for iPhone and Android applications using a coding-free graphical user interface. I will first describe the advantages and disadvantages of Tiggzi, and I will then share with you my recommendation about the service.

Advantages

  1. Tiggzi does not require any coding. Dragging and dropping widgets on a canvas is the only thing that is required from a user of Tiggzi.

  2. The applications created by Tiggzi can be be run on iPhones and iPads as well as on Android devices and mobile-phone browsers.

  3. Creating application screens is straightforward in Tiggzi, and transitioning between those screens is very easy. Also, each button, label, image can navigate to a new screen by selecting the appropriate settings.

  4. It is very easy for a user to define and connect to REST services. Importing data using JSON, for instance, is very easy and the results of the queries can be easily included in a table view. This common task is handled very well in Tiggzi and a tutorial on Tiggzi’s website explains the process step-by-step.

  5. The source code that Tiggzi generates is a collection of HTML and CSS files bundled with PhoneGap. That code can be edited in Apple’s XCode.

  6. The Tiggzi website contains several well-presented tutorials that describe the creation of data-driven applications, which are common in business.

  7. Tiggzi makes it very easy to share an application with testers and clients by supplying a custom URL with the HTML/CSS source code of the application. In the paid version, Tiggzi allows team members to collaborative on the creation of the applications.

  8. For Android applications Tiggzi provides an app that runs the code generated by Tiggzi natively on the phone, even before the application is launched and submitted in the Android marketplace.

Despite its great number of advantages, Tiggzi may not be the platform that some developers may choose. That’s because the service has a few disadvantages.

Disadvantages

  1. Tiggzi does not generate native GUI elements, at least for the iPhone that I tested it with. As a result, applications do not feel native. In addition, I had several problems using a table view, a common element in the business applications that I am interested in creating.

  2. The service is not free. Tiggzi has several subscription levels but the free version provides a great feel for the service. (The free tier is currently limited to one project and on constructing three screens.)

  3. A developer depends on the web-interface to create the application. That interface may change as a function of time. Although this might not be a disadvantage, provided that the interface improves.

Although Tiggzi has a lot of advantages, the lack of non-native GUI elements, might stop developers from using the service to create polished applications. However, Tiggzi provides a simple interface for creating apps and for sharing working prototypes with testers and clients.

  2. Chemistry-related calculators
  3. Books
  4. Reference works
  5. Games

These categories can be subdivided into additional categories, but for the purposes of this post, let’s stick with those.

Now imagine an iPhone application that has a tab bar at the bottom with two tabs: One is called “Featured” and the other one is called “Categories.” See the screenshot below. When you tap on one of the categories you can see a list of related apps, “App 1,” “App 2,” etc. When you tap on the “Featured” tab you can see a list of featured apps, “Featured App 1,” “Featured App 2,” etc.

When I started with iPhone development, I thought that creating such a simple app should be very easy. Well, I was in for a surprise: it was not easy at all. Mostly because Objective-C is not an easy language to program.

But recently I found a code generation platform that made developing such a simple application, well simple. The platform is called Canappi. In order to create an app in Canappi you need to use a specialized language called MDSL. Using MDSL one can easily create application menus, views, and layouts.

The MDSL code below demonstrates how straightforward it can be to create the iPhone app shown above.

First, you need to identify your app

package com.yourname.chemapps ;

Second, you need to create three layouts. One for the app list, one for the featured apps list and one for the list of categories.

For the app list layout, first you create a table that is going to be used as menu and you populate it with menu items. When each of the menu items is tapped, the application navigates to a screen with the details of the selected app.

layout appsLayout {
    table buttonMenu as menu {
        menuitem 'App 1' { action displayGradientButtons navigate to detailView ; }
        menuitem 'App 2' {action displayGradientButtons navigate to detailView ; }
        menuitem 'App 3' { action displayGradientButtons navigate to detailView ; }
        menuitem 'App 4' { action displayGradientButtons navigate to detailView ; }
        menuitem 'App 5' { action displayGradientButtons navigate to detailView; }
    }
}

The layout for the featured applications is the same as that for the app list, apart from the titles of the menu items.

layout featuredLayout {
    table buttonMenu as menu {
        menuitem 'Featured App 1' { action displayGradientButtons navigate to detailView ; }
        menuitem 'Featured App 2' { action displayGradientButtons navigate to detailView ; } menuitem 'Featured App 3' { action displayGradientButtons navigate to detailView ; } menuitem 'Featured App 4' { action displayGradientButtons navigate to detailView ; } menuitem 'Featured App 5' { action displayGradientButtons navigate to detailView ; } } }

Similarly for the layout of the various categories, but the target of each row is different: it’s the examPrepView when the “Exam Prep” option is selected, etc.

layout categoriesLayout {
    table buttonMenu as menu {
        menuitem 'Exam prep' { action displayGradientButtons navigate to examPrepView ; }
        menuitem 'Calculators' { action displayGradientButtons navigate to calculatorsView ; }
        menuitem 'Books' { action displayGradientButtons navigate to booksView ; }
        menuitem 'Reference' { action displayGradientButtons navigate to referenceView ; }
        menuitem 'Fun' {action displayGradientButtons navigate to funView ; }
        }
    }

// Empty view
}

view categoriesView 'Categories' {
    controls {
        layout categoriesLayout ;
    }
}

view featuredView 'Featured' {
    controls {
        layout featuredLayout ;
    }
}

Fourth, you need to look at the main window of the application, add a navigation bar, declare the view from which the application will start, as well as the items in the tab-bar menu.

main chemapps {
    navigationBar;
    start categoriesView ;
    menu { categoriesView , featuredView}
}

And that’s it. If you upload this file to Canappi’s code generator, you will receive a zip file with the XCode project file and source code for your application (in addition to the source code for the Android platform.) You can open those files in XCode, and run the compiled application.

Several notes are in order:

  1. MDSL files can be edited in Eclipse, the widely used integrated development environment, as described on Canappi’s website.

  2. The generation of XCode files is done only by uploading an MDSL file to canappi.com

  3. You have to become a member of the Canappi website in order to have access to the servers at canappi.com.

  4. The use of Canappi’s servers is currently free, but the developer has plans to charge for their use.

  5. If you create an interface in XCode’s interface builder, you can use a tool in canappi.com to create an MDSL file that you can then edit. At Canappi.com you can also import an interface created in Balsamiq, and export it to an MDSL file.

Canappi provides an alternative to mobile development if you are not very comfortable with Objective-C. With only little effort, simple applications can be easily created, not only for the iPhone but for Android too.

Read more about the project by visiting www.canappi.com.

Five Reasons Why You See More and More People With iPhones

Jan 16th, 2012

Do you see more and more people carrying iPhones these days? I certainly do: Everywhere I look, I see iPhones, new and old. So what do you think?

Are there more and more people using iPhones these days?

I set out to answer this question and here’s what I found:

  • 43% of the smartphones sold in October/November 2011 were iPhones, according to a report from TechChrunch.

  • 54% of the people surveyed who plan on buying a smartphone sometime between now and March, plan to buy an iPhone, according to a report from ChangeWave Research.

These facts along with the permanently customer-filled Apple stores, the news that a riot almost broke in Beijing the first day that iPhone 4S went on sale, and my constant noticing of people carrying iPhones, got me convinced: iPhones must be the most popular smartphones. That’s until I read the following report from Apple Insider:

Apple’s share of the smartphone market is 18%, below that of Samsung at 19%.

So?

How come there seem to be so many iPhones despite their 18% market share?

I am not a marketer or a consumer psychologist, but here are five reasons why you may notice more and more people with iPhones:

  1. There’s only one iPhone but hundreds of Android phones: Because there is only one iPhone, but hundreds of Samsung phones, it is easier to recognize an iPhone. This behavior might be related to what social theorist Barry Schwartz calls the paradox of choice: more choices are not necessarily better.

  2. Apple aggressively advertises its products: When Apple announces a new device, everybody knows about it: the media, the bloggers, everybody. So after such a barrage of information, when you see a new Apple device, you recognize it.

  3. Apple aggressively places its products in movies: Apple products are aggressively placed in popular movies, about 30% of them in 2010, according to Apple Insider. Apple products are placed more frequently in movies than those of Nike or Ford.

  4. The iPhone 4S was introduced in October: Happy owners of the new iPhone 4S were busy playing with their device, effectively displaying it more frequently to friends and strangers.

  5. The iPhone has become a very popular camera: At least for Flickr users, the iPhone is the most popular camera. And there is no easier way for people to display their iPhones than when they take it out to take a picture, especially during the holiday season.

So what do you think? How do you explain that there seem to be so many iPhones around despite their 18% market share? Share your opinion in the comments.

Getting Started With Developing iOS Apps: Hardware

Jan 7th, 2012

The first question that I had when I started with iPhone/iPad app development was what hardware was necessary for the job. Could I develop my apps on Windows or, even better, could I do it using my Linux machine? And would I need to buy an iPad in addition to the iPhone that I already owned?

After I did a lot of of research these are the answers that I found:

You need an Intel-based Mac. The alternatives are for the adventurous

If you google “hardware for iPhone development” you will find that the only hardware that is sanctioned by Apple to develop apps is an Intel-based Macintosh. That means that Apple does not make Windows or Linux versions of XCode, its integrated development environment (IDE) that is used for app development.

Given that I have been a Linux and Windows user for years, I thought that there must have been some alternatives for these operating systems. So I some did extensive research, and here’s the bottom line: In order to focus my energy on developing applications, and not on troubleshooting operating system issues, I needed an Intel-based Mac.

If you cannot afford to buy an Intel-based Mac however, there are a few alternatives. Here are the two about which I have learned a little bit about:

  1. If you are NOT interested in submitting your applications to the Apple app store, you don’t need XCode. In that case, you can compile and debug your apps with an alternative IDE, test them on a jailbroken iPhone, and submit them to Cydia, a repository of applications for jailbroken iPhones.

  2. If you are interested in submitting your applications to the Apple app store, you need XCode, and the Mac operating system (Mac OS). It is possible to install the Mac OS on non-Apple hardware, by using one of the following two methods, neither of which is sanctioned by Apple:

* You can put together a Hackintosh, a non-Apple computer that runs Mac OS. Only certain non-Apple computers can be "converted" into Hackintoshes.


* You can patch the commercial software VMware Workstation, and obtain a modified installation disk of Mac OS. Both actions may be legally questionable. But if you

choose to go that route, the non-Apple hardware and the host operating system (Windows or Linux) must supports virtualization.

You need to choose your Intel-based Mac carefully

You may consider buying a used Mac to save some money. That was my first thought when I found out the price of new Macs. But you have to be careful. You should make sure that the used computer runs the latest version of the operating system. The reason is that Apple has a “my way or the highway” attitude towards its app store and tightly controls app development. For instance, Apple makes the latest version of XCode run on only the latest two versions of Mac OS; And you do want the latest version of XCode because each new version makes app development easier.

You should also be careful about the size of the hard disk of your Mac. Flash-based disks are much faster than conventional ones but they are also much smaller for the price. If you are considering using your Mac as your main computer, you may need some additional space on your hard disk.

You need to have have an iPhone and/or an iPad

Although XCode comes with a iPhone/iPad simulator, it is extremely useful to have the actual iOS devices to test your applications on. If you are planning to develop apps for only the iPhone or only for the iPad, you don’t need to have both.

Relax and enjoy Apple’s tight control of app development

As I mentioned above, Apple tightly controls the hardware and the software that is necessary to develop apps for its mobile devices. In addition, not all apps that have been developed on Apple’s hardware and with Apple’s software are approved by Apple. It turns out for an app to get approved it has to follow strict guidelines for design and architecture.

Such control however, come with some advantages:

  • Apple is making available its innovative software development products to those who follow the company’s strict guidelines.

  • If your app is approved, there is very little that you have to do to market it. Apple does all the work for you, and the users will chose it if it’s worthwhile.

  • iPhone and iPad users already trust Apple to approve useful and beautiful apps. Therefore users are more willing to invest on such apps.

Finally, I have not been able to find anyone who claims to have developed an iPhone/iPad application by using a method that is not sanctioned by Apple.

So after a lot of thought and research, I begrudgingly bought a Mac. My Mac has now become my main work computer and because its operating system is based on Unix BSD, my transition from Linux has been very smooth. And, I almost hate to say it, I like my new Mac.

Have you developed apps that are featured in the App store without Apple’s sanctioned hardware and software? If yes, please share your experience in the comments.
